What is void elixir

As we know, in Elixir modules are not bound to source files in any way. This is a deliberate decision that plays well with Elixir’s rich macro support. However, it can lead to complicated code.

Void Elixir, or void.ex, is a convention that sets 1:1 binding between the module and the source file. Indirectly, it also discourages the excess relience on macros and promotes simplicity and clean code.

The convention is quite intuitive: “higher-order” modules are mapped to directories, and leaf modules - to *.ex files. The contents of a “higher-order” module should be placed in the void.ex file of its corresponding directory.

Let’s see an example. Suppose we have “The Three Little Pigs” project, which aims to help the pigs with various aspects of their life. Here is what the source files could be:

PigLife
├── House
│   ├── Brics.ex
│   ├── Sticks.ex
│   ├── Straw.ex
│   └── void.ex
├── Music
│   ├── Fiddle.ex
│   ├── Flute.ex
│   ├── Piano.ex
│   └── void.ex
└── void.ex

As you see, every directory contains leaf modules, subdirectories and the void.ex file. The name of the module the latter contains, is the path of the directory it resides in:

# PigLife/House/void.ex

defmodule PigLife.House do

alias PigLife.House.Straw
alias PigLife.House.Sticks
alias PigLife.House.Bricks

def build(laziness) do
  case laziness do
    5 -> Straw.build()
    4 -> Sticks.build()
    3 -> Bricks.build()
  end
end

end

Notice the lack of indent after defmodule? That’s the code style to use with void.ex.

Why is it called void?

I think you have figured that out by now: for a regular leaf-module, to get its name, you concatenate its directory path to its filename. For void.ex, on the other hand, you just take its directory path. Or, concatenate it with void.

Is this just a convention for organizing source files?

It is. No, not only that: also, the lack of indent after defmodule. Indeed, the indent is not needed anymore: otherwise, the whole module would be indented.

defmodule in Elixir gives a name to a module. Hovewer, with void.ex, the module name can be easily derived from its filename - thus, isn’t too valuable.

What’s the value in it?

This is the most interesting part.

  • Having multiple modules per file encourages using modules for things other than just holding functions, leading to all sorts of alchemy. Binding a module to a file, on the other hand, leaves less room for __using__ the compile-time magic.

  • Declaring “use” on a per-file basis will make it less hermetic and thus less attractive.

  • No indent after defmodule will help convey the idea that it should just say the module name, no more.

  • Last but not least, having a predictable file structure makes your code more transparent and easier to navigate through.

Modules should be just the containers of functions, like it is in Erlang! Void the insanity, simplicity is bliss!

Who is it for?

It’s for everybody who loves simplicity and clean code. But especially - for erlangers that start to write Elixir and are shocked by the amount of keywords and custom syntax. Keep calm and stick to the Erlang way!
